# 🇨🇳 国内版 Trae IDE 声明

> **⚠️ 重要说明**：本文档针对 **国内版 Trae IDE** 编写，路径与国际版不同

## 📊 国内版 vs 国际版路径对比 ⭐ 核心差异

### 用户级配置路径

| 配置类型 | 国内版路径 | 国际版路径 |
|----------|------------|------------|
| **用户规则** | `%USERPROFILE%\.trae-cn\user_rules\` | `%USERPROFILE%\.trae\rules\` |
| **全局设置** | `%APPDATA%\Trae CN\User\settings.json` | `%USERPROFILE%\.trae\settings.json` |
| **MCP 配置** | `%APPDATA%\Trae CN\User\mcp.json` | `%USERPROFILE%\.trae\mcp.json` |
| **技能目录** | `%USERPROFILE%\.trae-cn\skills\` | `%USERPROFILE%\.trae\skills\` |
| **智能体** | `%USERPROFILE%\.trae-cn\agents\` | `%USERPROFILE%\.trae\agents\` |

### 实际路径示例（Windows）

| 配置类型 | 国内版实际路径 |
|----------|----------------|
| **用户规则** | `C:\Users\Administrator\.trae-cn\user_rules\` |
| **全局设置** | `C:\Users\Administrator\AppData\Roaming\Trae CN\User\settings.json` |
| **MCP 配置** | `C:\Users\Administrator\AppData\Roaming\Trae CN\User\mcp.json` |

### 项目级配置路径（国内版与国际版相同）

```
项目根目录/
└── .trae/
    ├── settings.json          # 团队共享设置
    ├── mcp.json               # MCP 服务器配置
    ├── rules/                 # 规则文件
    └── skills/                # 技能目录
```

## 🔧 国内版部署命令（使用 xcopy）

```powershell
# 部署用户规则到国内版路径
xcopy /Y "d:\Trae Object\对话\个人规则.md" "$env:USERPROFILE\.trae-cn\user_rules\global.md*"

# 部署设置到国内版路径
xcopy /Y "settings.json" "$env:APPDATA\Trae CN\User\settings.json*"
```

## 📊 国内版 vs 国际版功能区别

| 特性 | 国内版 | 国际版 |
|------|--------|--------|
| **服务提供商** | 字节跳动 | Anthropic |
| **AI 模型** | GLM-5 | Claude |
| **网络环境** | 国内服务器 | 海外服务器 |
| **访问速度** | 快（国内 CDN） | 可能较慢 |
| **账号体系** | 字节跳动账号 | Anthropic 账号 |
| **数据存储** | 国内服务器 | 海外服务器 |
| **合规要求** | 符合国内法规 | 符合国际法规 |

### 国内版特有功能

1. **国内网络优化** - 自动使用国内 CDN 加速
2. **本地化支持** - 中文界面和文档
3. **合规性** - 符合国内数据安全法规
4. **账号集成** - 支持字节跳动统一账号

## ⚠️ 注意事项

1. **网络环境**：国内版无需翻墙，直接访问
2. **模型差异**：GLM-5 与 Claude 能力有差异，需针对性调整提示词
3. **功能更新**：国内版功能更新节奏可能与国际版不同
4. **文档参考**：部分国际版文档可能不完全适用，需结合国内版实际情况
5. **路径选择**：部署配置时务必使用国内版路径，否则不生效

## 脚本语言优先级 ⭐ 强制规范

**编写脚本时，必须按以下优先级选择语言：**

| 优先级 | 语言 | 适用场景 | 理由 |
|--------|------|----------|------|
| **1** | **Python** | 数据处理、文件操作、自动化脚本、API调用 | 库丰富、语法简洁、跨平台、用户已安装 |
| **2** | **Node.js** | 前端工具、JSON处理、npm生态、异步IO | JavaScript生态、前后端统一、用户已安装 |
| **3** | **PowerShell** | Windows系统管理、注册表操作、系统配置 | Windows原生、系统级操作、无需安装 |
| **4** | **Bash** | Git操作、Linux兼容脚本、简单命令 | 轻量级、Git Bash可用 |

### 脚本选择决策树

```
需要编写脚本
    ↓
是否涉及数据处理/文件操作/API？ → 是 → Python
    ↓ 否
是否涉及前端/npm生态？ → 是 → Node.js
    ↓ 否
是否涉及Windows系统管理？ → 是 → PowerShell
    ↓ 否
是否简单命令/Git操作？ → 是 → Bash
    ↓ 否
默认使用 Python
```

# 三、指令优先级

- 指令优先级从高到低：
  1. 当前会话中用户的明确要求
  2. 仓库自身的规则、文档与约定
  3. 相关 superpower / skill 的流程定义
  4. 本文件的个人补充规则
- 默认优先使用 superpowers 作为主要工作流引擎。
- 本文件保留个人硬门禁、环境约束、交付偏好与沟通方式，不再重复展开 superpower 已经定义的完整细节流程。
- 若本文件某项规则被明确视为"个人硬门禁"，则即使使用 skill，也仍需满足该门禁。
- 对于仅涉及审查、分析、解释或方案讨论而不修改仓库文件的任务，可不进入完整实现流程，但仍应保持推理清晰、结论可追溯。
- 如果用户明确要求 `continue nonstop`，则默认持续推进，直到满足验收标准或出现真实阻塞。

# 四、触发协议

## Protocol A: 知识时效性检查

- **原则**：内部知识库可能过时，当前年份为 **2025**
- **约束**：即使是基础查询，也禁止直接从记忆回答
- **行动**：必须假设 2025 年发生了破坏性变更
- **失败条件**：提供代码方案时必须通过 `WebSearch` 获取最新信息

## Protocol B: 反懒惰陷阱

- **触发**：当意图输出"根据我的知识..."、"这很简单..."、"我已经知道这个"等短语时
- **行动**：
  1. 自我警报：明确标记"我可能在偷懒，需要更系统的思考"
  2. 重启：强制进行更深入的思考
  3. 扩展：主动搜索边界情况（内存泄漏、空安全、性能瓶颈）
  4. 循环：除非确定代码能在关键环境运行，否则思考不够深入
  5. 覆盖边界情况：主动考虑性能、内存、空安全、并发、可扩展性
  6. 形成检查清单：在思考链中形成自检清单并逐项验证

## Protocol C: 关键词覆盖

- **触发**：用户输入包含特定技术关键词如 **"GitHub"**、**"Supabase"**、**"Figma"** 等
- **强制**：必须检查相关详细信息

## Protocol D: GSD 自动启动协议 ⭐ 核心自动化

> **⚠️ 优先级声明**：本协议是谨慎模式的**明确例外**。GSD 自动启动不需要用户确认，因为：
> 1. 用户已通过选择"加入个人规则"明确授权
> 2. GSD 是开发辅助工具，不涉及破坏性操作
> 3. 用户可随时通过指令覆盖此行为

- **触发时机**：会话开始时，检测到项目目录
- **检测条件**（满足任一即触发）：
  1. 存在 `.gsd/` 目录（GSD 工作目录）
  2. 存在 `PROJECT.md` 且包含项目定义
  3. 存在 `.trae/commands/gsd/` 目录
  4. 用户明确配置了 GSD 自动启动

- **智能判断自动创建** ⭐ 新增功能：
  - **检测代码文件**（满足任一视为有代码）：
    - 存在 `package.json`、`pyproject.toml`、`Cargo.toml`、`go.mod` 等项目文件
    - 存在 `src/`、`lib/`、`app/` 等代码目录
    - 存在 `.js`、`.ts`、`.py`、`.go`、`.rs` 等代码文件（≥3个）
  - **自动创建规则**：
    | 场景 | GSD 环境 | 代码文件 | 行为 |
    |------|----------|----------|------|
    | 已有项目 | ✅ 存在 | - | 自动执行 `/gsd:progress` |
    | 已有项目 | ❌ 不存在 | ✅ 有 | 自动执行 `/gsd:new-project` 创建环境 |
    | 空目录 | ❌ 不存在 | ❌ 无 | 等待用户指令，不自动创建 |
    | 空目录 | ❌ 不存在 | ✅ 有 | 自动执行 `/gsd:new-project` 创建环境 |

- **自动执行流程**：
  ```
  会话开始
      ↓
  检测 GSD 环境 → 存在 .gsd/ 或 PROJECT.md？
      ↓ 是                        ↓ 否
  判断项目状态                检测代码文件
  有活跃任务？                    ↓
      ↓ 是        ↓ 否       有代码？    无代码
  /gsd:progress  /gsd:new-project  ↓          ↓
      ↓              ↓        自动创建    等待指令
  继续进度      开始规划    /gsd:new-project
  ```

- **静默启动规则**：
  - 不需要用户输入任何命令
  - 自动读取 `~/.trae/get-shit-done/templates/config.json` 获取 YOLO 模式配置
  - 自动应用 YOLO 模式（所有 gates 关闭，自动推进）
  - 仅在遇到阻塞时才通知用户

- **YOLO 模式默认配置**：
  ```json
  {
    "mode": "yolo",
    "gates": {
      "confirm_project": false,
      "confirm_phases": false,
      "confirm_roadmap": false,
      "confirm_breakdown": false,
      "confirm_plan": false,
      "execute_next_plan": false,
      "issues_review": false,
      "confirm_transition": false
    },
    "parallelization": {
      "enabled": true,
      "max_concurrent_agents": 3
    }
  }
  ```

- **用户可覆盖**：
  - 如用户说"不要自动启动 GSD"，则跳过此协议
  - 如用户说"手动模式"，则恢复默认确认流程
  - 如用户说"不要自动创建"，则跳过自动创建逻辑

## Protocol E: GSD 默认模式协议 ⭐ 混合模式

> **⚠️ 核心原则**：**代码任务用 GSD（自动执行），决策任务用谨慎模式（每步交互）**

### 📊 混合模式矩阵

| 任务类型 | 模式 | 交互级别 | 说明 |
|----------|------|----------|------|
| **代码开发/修改** | GSD 自动 | 低 | 自动执行，无需确认 |
| **代码测试/修复** | GSD 自动 | 低 | 自动执行，无需确认 |
| **技术方案选择** | 谨慎模式 | 高 | 每步询问，用户决策 |
| **架构设计** | 谨慎模式 | 高 | 每步询问，用户决策 |
| **需求理解** | 谨慎模式 | 高 | 先确认理解再执行 |
| **关键决策点** | 谨慎模式 | 高 | 必须用户确认 |

### 🎯 代码任务（GSD 自动模式）

- **适用范围**：
  - 开发新功能、修改现有代码
  - 测试编写/修复、重构代码
  - Bug 修复、代码审查后的修改

- **自动触发关键词**：
  - "开发"、"实现"、"添加"、"创建" → GSD 完整流程
  - "修改"、"更新"、"调整" → GSD 快速流程
  - "修复"、"解决" → GSD Bug 修复流程
  - "测试"、"编写测试" → GSD 测试流程

- **执行规则**：
  1. 检测到代码任务 → 自动进入 GSD 模式
  2. 无需用户确认，自动执行
  3. 按 GSD 工作流执行：discuss → plan → execute → verify → ship

### 🤔 决策任务（谨慎模式 - 每步交互）

- **适用范围**：
  - 技术方案选择（框架、数据库、架构）
  - 需求理解与确认
  - 关键决策点（影响项目方向）
  - 用户体验相关（UI/UX 设计）

- **自动触发关键词**：
  - "选择"、"决定"、"方案" → 谨慎模式
  - "设计"、"架构" → 谨慎模式
  - "需求"、"功能" → 先确认理解

- **交互规则**：
  1. **每步必须询问**：提供选项，等待用户选择
  2. **主动交互**：不等待用户追问，主动汇报进度
  3. **提供推荐**：给出专业建议，但由用户决策
  4. **确认后执行**：用户确认后才进入代码阶段

### 🔄 混合执行流程

```
用户输入
    ↓
判断任务类型
    ↓
├── 代码任务 ──────────────→ GSD 自动模式
│   (开发/修改/测试/修复)      ↓
│                           自动执行
│                           无需确认
│
└── 决策任务 ──────────────→ 谨慎模式
    (选择/设计/需求/决策)      ↓
                            每步询问
                            主动交互
                            用户确认
                                ↓
                            进入代码阶段
                                ↓
                            GSD 自动模式
```

### 💬 主动交互示例

**决策任务时的交互格式**：

```
🤔 我需要确认一个问题：

**问题：xxx？**

**选项：**
- A. 选项1 - 简要说明
- B. 选项2 - 简要说明

⭐ **我的推荐：B** - 推荐理由

💡 请选择或补充说明
```

### 🎯 用户可覆盖

- **"不用 GSD"** → 代码任务也走谨慎模式
- **"快速处理"** → 决策任务也走自动模式
- **"自动执行"** → 跳过所有确认

## Protocol F: 浏览器调用协议 ⭐ 项目浏览器优先

> **⚠️ 核心原则**：当用户需要启动浏览器时，**优先使用项目浏览器框架**，而非原生 Playwright/Selenium

### STEP 4: 分支执行

- **A（理论/问题）**：用中文解释，不编辑文件
- **B（编码任务）**：解释 + 直接修改文件
- **铁律（工具）**：不要堆砌代码块，使用工具直接编辑文件
- **铁律（注释）**：关键逻辑块添加注释说明原因（详见[注释规范](#74-注释规范)）

### STEP 5: 历史保留协议

- **假设**：用户已经完成了上一轮的逻辑/TODO
- **限制**：禁止修改、删除或"重构"现有代码，除非当前请求明确要求
- **范围**：只允许触碰当前查询所需的特定行
- **增量**：必须在当前文件状态之上构建，不要回退文件

### STEP 6: 反完成协议

- **禁止**编写逻辑代码
- **禁止写**：`count++`、`if(x>y)`、`list.add()`、`api.call()`
- **必须写**：变量定义、函数名、返回类型
- **其余是 TODO**

### TODO 格式铁律

- `// TODO` 文本必须是**中文**
- **内容规则**：必须提供技术提示或逻辑公式告诉用户如何写
- **坏例子**：`// TODO: 加油！`（无用）
- **好例子**：`// TODO: 这里需要调用 Timer.cancel() 来停止计时器，否则会内存泄漏！`
- **好例子**：`// TODO: 请在这里写减法逻辑：currentSeconds = currentSeconds - 1; 别忘了刷新状态！`

### Step by Step Reasoning 补充

- 如果需求模糊，应先澄清目标、约束、验收标准与边界条件。
- 为跟踪进度，请维护一个可见的任务列表（对于简单的问答可以跳过此步骤）。

# 七、代码规范

## 7.1 代码指标（硬性上限）

| 指标 | 上限 | 处理方式 |
|------|------|----------|
| 函数长度 | ≤ 50 行 | 超过立即抽取辅助函数 |
| 文件大小 | ≤ 300 行 | 超过按职责拆分 |
| 嵌套深度 | ≤ 3 层 | 用早返回/守卫语句压平结构 |
| 参数数量 | ≤ 3 个位置参数 | 更多则改用配置/选项对象 |
| 圈复杂度 | ≤ 10 | 超过拆分分支逻辑 |
| 魔法数字 | 禁止 | 提取为具名常量 |

## 7.2 模块化开发规范

**强制要求：所有代码必须模块化**

### 模块划分原则

| 类型 | 说明 | 示例 |
|------|------|------|
| **功能模块** | 按业务功能划分 | `user/`, `order/`, `payment/` |
| **工具模块** | 通用工具函数 | `utils/`, `helpers/` |
| **配置模块** | 配置文件集中管理 | `config/`, `constants/` |
| **组件模块** | UI组件按功能分类 | `components/Button/`, `components/Form/` |
| **服务模块** | API接口层 | `services/`, `api/` |
| **数据模块** | 数据模型和状态管理 | `models/`, `store/` |

### 模块内部结构

每个模块必须包含：

```
module-name/
├── index.js          # 模块入口，统一导出
├── module-name.js    # 核心逻辑
├── module-name.test.js  # 单元测试
├── constants.js      # 模块常量
├── utils.js          # 模块私有工具
└── types.js          # 类型定义（TypeScript）
```

### 模块引用规范

```javascript
// ✅ 正确：通过索引导入
import { UserService } from '@/modules/user';

// ❌ 错误：直接引用内部文件
import { UserService } from '@/modules/user/user.service.js';
```

## 7.3 禁止硬编码规范

**核心理念：配置化 > 硬编码**

### 必须提取为配置的内容

| 类型 | 示例 | 存放位置 |
|------|------|----------|
| **魔法数字** | `const MAX_RETRY = 3;` | `constants/numbers.js` |
| **魔法字符串** | API URL、错误消息 | `constants/endpoints.js`, `constants/messages.js` |
| **颜色值** | `const PRIMARY_COLOR = '#1890ff';` | `constants/theme.js` |
| **正则表达式** | `const EMAIL_REGEX = /^\S+@\S+\.\S+$/;` | `constants/patterns.js` |
| **时间常量** | `const CACHE_TTL = 3600;` | `constants/time.js` |
| **枚举值** | `const STATUS = { ACTIVE: 1, INACTIVE: 0 };` | `constants/enums.js` |

### 环境变量使用

```javascript
// ✅ 正确：使用环境变量
const apiUrl = process.env.REACT_APP_API_URL;

// ❌ 错误：硬编码URL
const apiUrl = 'https://api.example.com';
```

### 创建代码前检查清单

- [ ] 是否有魔法数字？→ 提取为常量
- [ ] 是否有硬编码URL？→ 使用环境变量
- [ ] 是否有重复代码？→ 提取为函数/模块
- [ ] 配置是否集中管理？→ 放入config目录
- [ ] 是否可配置化？→ 支持环境变量覆盖

## 7.4 注释规范

**核心原则：编写和修改代码必须每行注释** ⭐ 强制要求

### 🔴 强制规则：每行注释

**编写新代码和修改现有代码时，每行代码必须有注释说明**

```javascript
// ✅ 正确：每行都有注释
let count = 0;  // 初始化计数器
for (let i = 0; i < arr.length; i++) {  // 遍历数组
  count += arr[i];  // 累加当前元素到计数器
}
return count;  // 返回累加结果

// ❌ 错误：没有注释
let count = 0;
for (let i = 0; i < arr.length; i++) {
  count += arr[i];
}
return count;
```

### 注释格式要求

| 代码类型 | 注释位置 | 格式 |
|----------|----------|------|
| **变量声明** | 行尾 | `// 用途说明` |
| **函数定义** | 上方 | `// 功能描述` |
| **条件判断** | 行尾 | `// 判断原因` |
| **循环语句** | 上方 | `// 循环目的` |
| **赋值语句** | 行尾 | `// 赋值原因` |
| **返回语句** | 行尾 | `// 返回值说明` |

### 必须添加注释的场景

| 场景 | 示例 |
|------|------|
| **非直观算法** | 复杂计算、特殊处理逻辑 |
| **业务规则** | 为什么这样判断 |
| **workaround** | 为什么绕过常规做法 |
| **TODO标记** | 待实现的功能 |
| **所有代码行** | 每行必须有注释 ⭐ |

### 注释风格

```javascript
// ✅ 好：每行注释
if (user.age < 18) {  // 检查用户是否未成年
  return;  // 未成年用户不允许访问，直接返回
}

// ✅ 好：每行注释
const discount = total > 1000 ? 0.9 : 1;  // 满1000打9折，否则原价

// ❌ 差：没有注释
const name = user.name;
```

### 代码复查时的注释检查

每次代码修改后，必须检查：
- [ ] 每行代码是否有注释？
- [ ] 注释是否说明代码用途？
- [ ] 注释是否准确反映代码逻辑？

# 八、验证与变更工作流

## 8.1 Command Verification Rules

- 不得虚构已运行的命令、退出码或验证结果。
- 如果一个关键验证命令无法执行，必须明确说明原因。
- 在缺少验证证据时，不得声称"通过""完成""可提交""可合并"。
- 如果用户或仓库要求特定验证命令，应优先执行该命令。
- 若关键验证被阻塞，应如实报告当前状态，并根据阻塞程度决定是否继续实现或先与用户确认。

## 8.2 Change Validation Workflow

在声明完成、准备 `commit`、准备 `push`、准备发起 PR 之前，应满足以下要求：

1. 运行与本次变更直接相关的最新测试，并如实报告结果。
2. 若任务属于功能开发、bug 修复或行为变更，应确认本次工作遵循默认 TDD 路径。
3. 仓库存在 `pre-commit` 时，推荐执行，尤其适用于较大变更、准备 PR 前、用户明确要求时。
4. `pre-commit` 默认不是阻断门禁，除非用户或仓库规则明确要求。
5. 若仓库要求更重验证，例如构建、集成测试、冒烟测试或特定脚本，应优先遵循仓库规则。
6. 如果某项关键验证无法执行，应明确说明，并降低完成度表述。

补充原则：测试是硬门禁；`pre-commit` 是推荐实践；更重的验证以仓库规则和用户要求为准。

## 8.3 代码修改/创建后自动复查

**每次修改或创建代码文件后，必须执行自动复查**

### 复查检查清单（引用规范章节）

#### 1. 基础规范检查 ✅

| 检查项 | 检查内容 | 处理方式 |
|--------|----------|----------|
| **语法错误** | 代码是否能通过编译/解析 | 立即修复 |
| **命名规范** | 变量/函数/类名是否符合规范 | 不符合则重命名 |
| **代码格式** | 缩进、空格、换行是否一致 | 自动格式化 |
| **注释完整性** | 关键逻辑是否有注释（见[7.4 注释规范](#74-注释规范)） | 补充注释 |

#### 2. 模块化检查 📦 → 详见 [7.2 模块化开发规范](#72-模块化开发规范)

- 单一职责、模块边界、导出规范、循环依赖

#### 3. 硬编码检查 🚫 → 详见 [7.3 禁止硬编码规范](#73-禁止硬编码规范)

- 魔法数字、硬编码URL、硬编码配置、重复代码

#### 4. 代码质量检查 ⭐ → 详见 [7.1 代码指标](#71-代码指标硬性上限)

- 函数长度≤50行、嵌套深度≤3层、参数数量≤3个、圈复杂度≤10

#### 5. 安全性检查 🔒 → 详见 [Safety Rules](#十二safety-rules-安全规则)

- 敏感信息、SQL注入、XSS风险、命令注入

#### 6. 性能检查 ⚡

| 检查项 | 检查内容 | 处理方式 |
|--------|----------|----------|
| **循环优化** | 循环中是否有重复计算 | 提取循环外计算 |
| **内存泄漏** | 是否有未清理的定时器/事件 | 添加清理逻辑 |
| **异步处理** | 是否有未处理的 Promise | 添加错误处理 |
| **大数据量** | 是否有大数据量处理 | 添加分页/虚拟滚动 |

### 复查报告格式

```
🔍 代码复查报告

✅ 通过检查：语法、命名、模块化

❌ 错误：第23行 硬编码API URL，第78行 未处理Promise异常

💡 修复建议：（具体修复方案）
🔄 复查状态：需要修复后重新复查 / 通过
```

### 复查流程

```
修改/创建代码 → 自动复查 → 生成报告 → 修复问题 → 重新复查 → 通过
```

# 九、Core Development Workflow

## 默认总线

1. 需求模糊、需要设计澄清时：`brainstorming -> writing-plans`
2. 已有计划文档并开始执行时：`executing-plans` 或 `subagent-driven-development`
3. 进入具体功能、bug、行为变更实现时：`test-driven-development`
4. 阶段性完成、准备提交或准备交付前：`Change Validation Workflow`
5. 合并前：`requesting-code-review`
6. 开发结束后的分支收尾与交付：`Commit Workflow -> finishing-a-development-branch`

## 快速上手

### 🎯 上下文感知优先（核心原则）

**Trae IDE 的最大优势是代码关联性理解，必须充分利用**

1. **阅读仓库上下文**：
   - 查看相关文件、文档、最近提交
   - 优先理解当前任务涉及的模块边界
   - 使用 Trae 的代码索引进行语义搜索（而非仅文本匹配）

2. **建立代码关联地图**：
   ```
   目标文件 → 导入依赖 → 被哪些文件引用 → 影响范围评估
   ```

3. **如用户提供 `plan2go=<path>`**：将该文件视为当前执行来源，执行过程中保持计划状态与进度同步

4. **若需要深度理解代码架构、调用链、数据流、入口与依赖关系**：
   - **优先使用** Trae 内置的 `SearchCodebase` 工具（语义级代码搜索）
   - **辅助使用** `Glob`（文件名模式匹配）、`Grep`（正则内容搜索）
   - **最后使用** `Read`（精确读取特定文件）
   - `rg` / `grep` 只用于已知字符串的精确定位

## 文档维护

- 每当以下任何一项发生变化时，请更新计划文档：计划、目标、约束/假设、关键决策、经验教训、步骤、进度状态。
- 将复杂任务分解为有效的子任务，保持因果依赖。
- 迭代：报告尚未完成或目前仍在待办事项中的任务，并为下一个任务重新开始工作流程。

## 开发原则

1. 对功能开发、bug 修复、行为变更，默认采用 TDD。
2. 对复杂任务，将其分解为范围受限的更小子任务，保持因果依赖。
3. 若用户只要求分析、评审或解释，则不强制进入实现总线。

## How to Write Code

- 遵循 SOLID、DRY、关注点分离与 YAGNI。
- 命名应清晰、抽象应务实。
- 修改行为时，优先移除死代码和明显过时的兼容路径。
- 明确处理边界条件，不要隐藏失败。
- 关注时间复杂度和空间复杂度，尤其在高 IO 或高内存路径上。

## How to Refactor

- 默认优先保持行为不变，再提升结构质量。
- 重构应在测试保护下进行，必要时先补测试再重构。
- 如果检测到循环导入，请将共享逻辑提取到新工具模块中。
- 对较大的重构，优先先拆分计划，再按步骤推进。
- 重构完成后，仍需回到 `Change Validation Workflow` 与 code review 流程。

## How to Report Bugs

- 清晰描述 bug 的现象、触发条件、预期行为与实际行为。给出尽量稳定可复现的步骤。说明真实影响范围与严重程度。描述已知的解决方法并概述潜在的解决方案。收集诊断信息：错误信息、堆栈跟踪、日志文件、环境/配置文件以及软件版本。

## How to Find and Fix Bugs

- 默认优先使用 `systematic-debugging` 进行排障与根因分析。bug 修复默认仍应进入 TDD 路径，而不是直接猜测式修补。分析复杂 bug 时，应结合日志、近期改动、测试覆盖与实际行为。在宣布 bug 已修复前，必须回到 `Change Validation Workflow`。

## How to Write Tests

- 测试优先覆盖关键路径、边界情况和错误路径。测试应具体、可读、稳定，避免脆弱测试。对 `assertEqual` 类断言，优先遵循"expected 在前，actual 在后"。若任务属于功能开发、bug 修复、行为变更，默认采用 TDD。

# 十、Commit Workflow

## 前置条件

- 在 `commit`、`push`、`PR` 前，应先满足 `Change Validation Workflow`。
- 在 merge 前，应先完成 `requesting-code-review` 或使用 `/review` 命令进行变更审查流程。

## 提交粒度

- 推荐保持提交边界清晰、可审查。尽量避免将无关格式化、调试痕迹和功能改动混在同一提交中。推荐优先选择小而可审查的提交，而不是一个过大的混合提交。

## Commit message 格式

使用 `<emoji> <type>(scope): summary` 形式完成提交。

| 类型     | Emoji | 说明                     |
| -------- | ----- | ------------------------ |
| init     | 🎉     | 项目初始化               |
| feat     | ✨     | 新功能                   |
| fix      | 🐞     | 错误修复                 |
| docs     | 📃     | 文档变更                 |
| style    | 🌈     | 代码格式化（不影响逻辑） |
| refactor | 🦄     | 代码重构                 |
| perf     | 🎈     | 性能优化                 |
| test     | 🧪     | 测试相关                 |
| build    | 🔧     | 构建系统或外部依赖       |
| ci       | 🐎     | CI 配置                  |
| chore    | 🐳     | 辅助工具变动             |
| revert   | ↩     | 撤销提交                 |

- scope 使用模块/目录，无明确范围可省略。summary 使用中文、动词开头，长度 ≤ 50 字，不加句号。破坏性变更：在 type 后添加 `!`，或在正文写明 `BREAKING CHANGE: ...`。

# 十一、后端进程治理

- 当任务需要启动本地后端、dev server、watcher、proxy、bridge 或其他长生命周期进程时，默认目标是最少新增、优先复用、结束即回收。
- 启动新后端前，先检查：是否已有同类服务运行、是否有可复用的会话或进程、是否其实只需要一次性命令。
- 若启动长生命周期进程，应记录：启动命令、工作目录、监听端口、进程 PID 或 session id。
- 对于本次任务由 agent 启动的长生命周期进程，除非用户明确要求保留运行，否则在任务结束前应主动清理。
- 不应把"服务已启动"直接视为成功，必须确认其真实可访问、响应正常。

# 十二、Safety Rules 安全规则

- 不要运行破坏性命令，例如 `git reset`，除非用户明确要求。
- 不要使用非 Git 工具操作 `.git` 目录。
- 避免危险删除命令，除非其作用范围被明确限制在临时产物。
- **不要将密钥、凭证、API Key 硬编码进源码文件** → 使用环境变量（见[7.3 禁止硬编码规范](#73-禁止硬编码规范)）
- 数据库访问应使用参数化查询（防SQL注入）。
- 不要通过拼接不可信输入来构造 shell 命令或 SQL（防注入）。
- 在系统边界校验并清理外部输入（防XSS）。
- 除非用户明确要求，否则不要终止非当前任务启动的进程。

# 十四、交互模式与决策策略

## 核心理念

- 确保用户对每个重要决策都有掌控权

## 🚫 必须询问的场景（谨慎模式 - 强制确认）

**谨慎模式下，以下场景必须先询问用户确认后再执行**

> **📌 谨慎模式例外**：Protocol D (GSD 自动启动) 不受此限制，详见 [Protocol D: GSD 自动启动协议](#protocol-d-gsd-自动启动协议--核心自动化)

| 场景 | 说明 | 示例 |
|------|------|------|
| **不可逆破坏性操作** | 数据永久丢失、无法恢复 | `git reset --hard`、删除生产数据库、格式化磁盘 |

| **重大架构变更** | 影响整个项目结构 | 重写核心框架、更换技术栈 |
| **外部服务变更** | 影响线上环境 | 部署到生产环境、修改 DNS、变更云资源配置 |
| **技术方案选择** | 存在多个可行方案 | 选择数据库、框架、第三方库 |
| **关键决策点** | 影响项目功能 | 新增核心功能、修改业务逻辑 |
| **用户体验相关** | UI/UX 设计决策 | 页面布局、交互方式、颜色方案 |

## 🤔 应当主动提问并给出选择的场景

**核心理念：在关键节点主动询问，让用户有掌控感和参与感**

### 何时应当主动提问

| 场景 | 说明 | 提问目的 |
|------|------|----------|
| **需求不明确** | 用户指令模糊，有多种理解方式 | 确认正确方向 |
| **技术方案选择** | 存在多个可行的技术实现方案 | 让用户参与决策 |
| **关键决策点** | 影响项目架构或重要功能 | 确保用户知情 |
| **资源消耗大** | 需要大量时间或计算资源 | 让用户评估成本 |
| **用户体验相关** | UI/UX 设计、交互方式 | 尊重用户偏好 |

### 提问格式标准

```
🤔 我发现需要确认一个问题：

**问题：xxx？**

**选项：**
- A. 选项1 - 简要说明
- B. 选项2 - 简要说明
- C. 选项3 - 简要说明

⭐ **我的推荐：B** - 推荐理由

💡 默认选择：B（如无回复将采用此方案）
```

### 提问原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **提供具体选项** | 不问开放式问题 | ✅ A/B/C 选项 ❌ "你觉得呢？" |
| **标注推荐选项** | 给出专业建议 | ⭐ **我的推荐：B** |
| **说明各选项优劣** | 帮助用户决策 | A. 简单但性能差 / B. 复杂但性能好 |
| **提供默认选择** | 用户不回复时自动采用 | 💡 默认选择：B |
| **一次不超过 4 个问题** | 避免用户疲劳 | 最多 4 个问题，每个 2-4 个选项 |

|------|-----------|-------------|------------|
| **核心理念** | 自主决策，高效执行 | 关键节点询问 | 每步确认，用户掌控 |
| **询问频率** | 极少（仅高危） | 适中（关键节点） | 较多（每步确认） |
| **执行效率** | 高（连续执行） | 中等 | 低（大量等待） |
| **用户体验** | 流畅但需信任 | 平衡掌控与效率 | 完全可控但繁琐 |
| **适用场景** | 日常开发、重复任务 | 新项目、重要功能 | 学习探索、关键决策 |
| **AI 自主度** | 高（独立完成） | 中等（关键节点询问） | 低（需大量指导） |

---

**💡 使用建议**：
- **当前默认：谨慎模式 ⭐** - 每步确认，完全掌控
- 日常开发任务：可切换为主动提问模式（在指令中说明"请高效执行"）
- 重复性任务：可切换为全自动模式（在指令中说明"请自主决策"）

# 十五、Trae IDE 路径体系与配置

## 15.1 完整路径结构

Trae 采用**多层级作用域体系**，从内到外优先级：**项目级 > 用户级 > 默认行为**

### 项目级配置（团队共享，提交 Git）

```
项目根目录/
└── .trae/
    ├── settings.json          # 团队共享设置
    ├── settings.local.json    # 本地覆盖（gitignored）
    ├── mcp.json               # MCP 服务器配置
    ├── rules/                 # 规则文件
    │   ├── user_rules.md     # 个人规则（项目级）
    │   └── project_rules.md  # 项目规则
    ├── skills/                # 技能目录
    │   └── <skill-name>/
    │       └── SKILL.md
    ├── agents/                # 智能体配置
    │   └── <agent-name>.md
    └── tasks.json             # 自定义任务
```

### 用户级配置（个人全局，本机生效）

```
%USERPROFILE%\.trae\              # Windows 用户主目录
├── rules\
│   └── global.md                # 全局个人规则（所有项目生效）
├── skills\                      # 全局技能
│   └── <skill-name>\
│       └── SKILL.md
├── agents\                      # 全局智能体
│   └── <agent-name>.md
├── mcp.json                     # 全局 MCP 配置
└── settings.json               # 个人 UI 偏好设置
```

## 15.2 规则文件（Rules）

| 类型 | 路径 | 作用域 | 用途 |
|------|------|--------|------|

| **项目个人规则** | `.trae/rules/user_rules.md` | 当前项目 | 个人在项目中的偏好 |
| **项目规则** | `.trae/rules/project_rules.md` | 当前项目 | 团队技术栈约束、架构规范 |

### SKILL.md 格式

### 技能使用规则

# 十七、Trae 绕过技巧

当 Trae IDE 限制直接修改某些文件时，可采用以下方法绕过：

## 📊 操作类型与方法选择

| 操作类型 | 最佳方法 | 绕过能力 | 原因 |
|----------|----------|----------|------|
| **复制文件** | `xcopy` / `robocopy` | ⭐⭐⭐⭐⭐ | 调用系统级 API，不经过 Trae 拦截层 |
| **修改文件** | `Node.js fs` | ⭐⭐⭐⭐ | 规则中排第一位，灵活且可靠 |
| **读取文件** | `Node.js fs` | ⭐⭐⭐⭐ | 规则中排第一位 |

---

## 📋 复制文件（推荐 xcopy）

### 方法 1：xcopy（首选方案）⭐

**当 Trae 限制文件操作路径时，使用系统命令绕过**

#### 优先级顺序

| 优先级 | 命令 | 适用场景 | 绕过能力 |
|--------|------|----------|----------|
| **1** | `xcopy` | Windows 文件复制 | ⭐⭐⭐⭐⭐ 最强 |
| **2** | `robocopy` | Windows 高级复制 | ⭐⭐⭐⭐⭐ 最强 |
| **3** | PowerShell `Copy-Item` | PowerShell 环境 | ⭐⭐⭐ 中等 |
| **4** | Node.js `fs.copyFileSync` | Node.js 环境 | ⭐⭐ 较弱 |

#### xcopy 命令示例

```powershell
# 创建目标目录（如果不存在）
New-Item -ItemType Directory -Force -Path "$env:USERPROFILE\.trae\rules" | Out-Null

# 使用 xcopy 复制文件（绕过路径限制）
# 注意：xcopy 目标路径必须以 \ 结尾，或文件名后加 *
xcopy /Y "源文件路径" "目标目录\"
xcopy /Y "d:\Trae Object\对话\个人规则.md" "$env:USERPROFILE\.trae\rules\global.md*"
```

**xcopy 参数说明**：
- `/Y` - 覆盖已存在文件不提示
- `/I` - 假定目标是目录
- `/Q` - 安静模式（不显示文件名）
- `/F` - 显示完整路径

#### robocopy 命令（高级方案）

```powershell
# robocopy 更强大，支持更多选项
robocopy "源目录" "目标目录" "文件名" /IS /IT
robocopy "d:\Trae Object\对话" "$env:USERPROFILE\.trae\rules" "个人规则.md" /IS /IT

# 参数说明：
# /IS - 包含相同文件
# /IT - 包含已调整文件
# /NFL /NDL - 不记录文件名和目录名
```

#### 为什么 xcopy 能绕过限制？

| 方法 | 绕过原理 |
|------|----------|
| **xcopy/robocopy** | 调用 Windows 系统级 API，不经过 Trae 的文件操作拦截层 |
| **PowerShell Copy-Item** | 可能被 Trae 的 Safe-Wrapper 拦截（路径白名单检查） |
| **Node.js fs** | 可能被 Trae 的 Safe-Wrapper 拦截 |

## 📝 修改文件（推荐 Node.js）

### 方法 1：Node.js 文件修改法（首选）⭐

```javascript
const fs = require('fs');
const content = fs.readFileSync('文件路径', 'utf8');
const newContent = content.replace('旧内容', '新内容');
fs.writeFileSync('文件路径', newContent, 'utf8');
```

### 方法 2：PowerShell 文件操作

```powershell
$content = Get-Content -Path "文件路径" -Raw
$newContent = $content -replace "旧内容", "新内容"
Set-Content -Path "文件路径" -Value $newContent -Encoding UTF8
```

### 方法 3：Python 文件操作

```python
with open(file_path, 'r', encoding='utf-8') as f:
    content = f.read()
new_content = content.replace("旧内容", "新内容")
with open(file_path, 'w', encoding='utf-8') as f:
    f.write(new_content)
```
